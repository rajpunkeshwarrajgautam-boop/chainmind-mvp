# Render Blueprint (`render.yaml`) — sync, Redis, Postgres

The repository root **`render.yaml`** defines a **Key Value** instance (Redis protocol) and a **Python web** service with `REDIS_URL`, `CELERY_BROKER_URL`, and `CELERY_RESULT_BACKEND` wired from that instance. `CELERY_TASK_ALWAYS_EAGER=true` keeps forecast tasks in-process (no separate worker service in this MVP).

What **cannot** be automated from this repository is finishing the service in the Render **dashboard**. Use this checklist after each meaningful change to `render.yaml` or to env wiring.

## 1) Create or attach the Blueprint

1. In [Render Dashboard](https://dashboard.render.com), open **Blueprints** (or **New** → **Blueprint**).
2. Connect the GitHub repo and select branch **`main`** (or your deploy branch).
3. Confirm the detected **`render.yaml`** path is the repository root.

## 2) Sync / apply so live services match the file

1. Open the Blueprint for **chainmind-mvp** (or your project name).
2. Use **Sync** / **Apply** (wording varies) so Render reconciles services with the current `render.yaml`.
3. If you renamed resources or changed types (for example adding Key Value), Render may prompt to **create** new resources or **update** existing ones — read the diff and confirm.

After sync:

- A **Key Value** service named **`chainmind-kv`** (or the `name:` in `render.yaml`) should exist.
- The **Web Service** should show env vars **`REDIS_URL`**, **`CELERY_BROKER_URL`**, **`CELERY_RESULT_BACKEND`** sourced from that Key Value **connection string** (not empty, not placeholder).

## 3) Set `DATABASE_URL` (required for real production)

`render.yaml` declares **`DATABASE_URL`** with **`sync: false`**. Render does **not** fill that from the file; you must set it in the UI.

1. Open the **Web Service** (API) → **Environment**.
2. Add or edit **`DATABASE_URL`** to your **managed Postgres** URL, in the form the app expects, for example:

   `postgresql+psycopg2://USER:PASSWORD@HOST:PORT/DATABASE`

3. Use the **Internal** URL from [Render Postgres](https://render.com/docs/postgresql) when both API and DB are on Render (lower latency, no public exposure).
4. **Redeploy** the service after saving env vars.

If you later split a **background worker** from the web process, set the **same** `DATABASE_URL` (and the same Redis/Celery URLs) on the worker service so it shares the database and broker.

## 4) Smoke-check

- **`GET /health`** on the public API URL should return `{"status":"ok",...}`.
- **`GET /ready`** may still report Redis/DB components depending on env; fix missing vars until it matches your expectations.

## 5) MVP behaviour (unchanged by design)

- **Stripe / OIDC:** routes return **501** until the corresponding secrets and issuer URLs are set (see `.env.example`).
- **Integrations / connectors:** template and audit flows until real credentials and sync are implemented — not blocked by Render wiring.
- **No “magic complete”:** the app does not invent API keys, Stripe accounts, or a database; you supply **`DATABASE_URL`** and secrets in the dashboard or secret store.

## 6) Alembic vs `create_tables()`

The API still runs **`create_tables()`** on startup. The revision **`20260426_0002_initial_schema`** only runs DDL when the database has **no** `tenants` table yet (for example, CI or a fresh Postgres where you run `alembic upgrade head` before boot). Existing deployments that already created tables via the app **do not** duplicate DDL.

For day-to-day schema changes after MVP, prefer new Alembic revisions (often autogenerated against a throwaway DB that matches production dialect) and review ops before merge — see `CONTRIBUTING.md`.
